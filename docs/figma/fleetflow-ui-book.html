01 - Charte graphique

Direction visuelle

FleetFlow doit ressembler à une application de bureau logistique : structurée, fiable, rapide à scanner, avec une identité sombre et bleue sans tomber dans le marketing décoratif.

Source: app actuelle

Personnalité

Interface opérationnelle B2B : robuste, calme, précise. Les effets visuels servent la hiérarchie, pas la décoration.

Hiérarchie

Titres lourds, labels uppercase courts, actions bleues franches. Les contenus longs restent en texte normal pour éviter la fatigue visuelle.

Sobriété

Ombres réduites, gradients discrets, cartes nettes. Les barres colorées sont réservées aux statuts ou priorités fortes.

05 - Formulaires

Lisibles, validés, explicites

Les erreurs API sont traduites côté UI. Un champ requis affiche toujours "Champ obligatoire" près du champ et dans le résumé d'erreur si le formulaire est complexe.

Champ obligatoire

Questions et scoring

  • Libre : réponse texte, score manuel sur 10, bonne réponse indicative possible.
  • Oui / Non : bonne réponse obligatoire, score auto 0 ou 10, override autorisé.
  • QCM : plusieurs choix, une ou plusieurs bonnes réponses, score proportionnel sur 10.
  • Note : échelle configurable, normalisée ensuite dans le total sur 100.
  • Éliminatoire : échec explicite si la réponse ou le score ne respecte pas le seuil.

06 - Workflows MVP

Écrans à respecter

Les parcours restent découpés en écrans clairs : listes, détails, édition, lancement, validation. Les actions critiques ne doivent pas être cachées dans des menus ambigus.

Admin / RH / Direction

Dashboard, contacts, jobs, tests, templates, pools, réponses et validations.

Global scope

Manager

Dashboard, tests assignés, formulaire par section, historique, sauvegarde brouillon et soumission.

Assigned only

Candidat

Offres publiques sans login, apply protégé, candidatures, tests assignés et profil.

Self scope

Détail test admin

Sections empilées en accordéon. Un clic ouvre les réponses, scores, commentaires et bonnes réponses; un second clic referme. Le feedback global arrive sous la dernière section.

Relancer un test

Bandeau titre compact, espace clair avant recherche, filtres visibles, cards respirées, barre d'action collée au bas de viewport sans masquer le dernier élément.

07 - RGAA AA renforcé

Règles non négociables

L'accessibilité est intégrée au design system : pas une passe cosmétique à la fin.

À faire

  • Utiliser de vrais <button>, <a>, <label>.
  • Relier chaque erreur au champ avec aria-describedby.
  • Garder un focus visible sur tous les contrôles.
  • Tester clavier : Tab, Shift+Tab, Enter, Escape sur modales.
  • Limiter les listes visibles à 5 éléments puis scroll interne clair.

À éviter

  • Texte gris trop faible sur surfaces sombres.
  • Actions uniquement en icône sans libellé accessible.
  • Styles inline ou handlers inline incompatibles CSP.
  • Cartes trop grandes qui étirent l'app desktop.
  • Scroll bloqué par une bottom nav ou une action bar.

08 - Code shared

Contrats à réutiliser

Les noms ci-dessous doivent rester les fondations des pages. Quand un écran a besoin d'une variante, l'ajout se fait dans shared avant la page.

  <section class="ff-app-container">

Décision

Une page peut composer des blocs, mais ne doit pas redéfinir les tokens, les boutons, les champs, les cartes ou les layouts principaux.

Évolution

Si un composant devient répété dans deux pages, il remonte dans libs/shared avec variante, état loading, état empty et état error.
